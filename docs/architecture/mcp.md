# MCP Server Architecture

**Date:** 2026-03-29
**Pipelines:** WORK-004-MCP-Server (#4), WORK-008-MCP-Resources (#8), WORK-009-Architecture-Doc (#9)

## Decision

MCP server via the `rmcp` crate (v1.3), feature-gated behind the `mcp` Cargo feature (which implies `storage`). The server exposes ScorchKit's scan engine, project management, finding lifecycle, scheduling, AI analysis, and browsable resources over stdio transport. CLI remains the direct-use interface; MCP is the AI-native interface.

## Rationale

- **MCP over REST API**: MCP is the protocol Claude Code and other AI assistants speak natively. A REST API would require a separate HTTP server, authentication, CORS, and clients would need custom integration. MCP gives us conversational tool use for free.
- **rmcp over hand-rolled JSON-RPC**: rmcp provides `#[tool]` macros for automatic JSON Schema generation, `ServerHandler` trait with typed dispatch, and transport abstractions. Hand-rolling JSON-RPC would be significant undifferentiated effort.
- **Feature-gated**: The `mcp` feature implies `storage` because every MCP deployment needs persistence (projects, scans, findings). Users who only want the CLI scanner don't need rmcp or sqlx.
- **stdio transport**: Claude Code communicates with MCP servers over stdin/stdout. No network configuration, no ports, no TLS. The server starts with `scorchkit serve` and runs until the client disconnects.

## Architecture

```
         ┌──────────────┐         ┌──────────────────┐
         │   CLI (clap) │         │   MCP Server     │
         │   args.rs    │         │   (rmcp stdio)   │
         └──────┬───────┘         └──────┬───────────┘
                │                        │
                │  Commands              │  Tools (24) + Resources (6) + Prompts (5)
                │                        │
         ┌──────▼───────┐        ┌───────▼──────────┐
         │   Runner     │        │ ScorchKitServer   │
         │  runner.rs   │        │   server.rs       │
         └──────┬───────┘        └───────┬──────────┘
                │                        │
                └────────┬───────────────┘
                         │
                         ▼
          ┌──────────────────────────┐
          │      Orchestrator        │    ← shared scan engine
          │    orchestrator.rs       │
          └──────────┬───────────────┘
                     │
          ┌──────────▼───────────────┐
          │      Storage Layer       │    ← shared persistence
          │   (PostgreSQL / sqlx)    │
          └──────────────────────────┘
```

Both the CLI and MCP server converge on the same Orchestrator for scanning and the same Storage layer for persistence. No logic is duplicated — MCP tools delegate to the same functions the CLI handlers call.

## Server Structure

### `ScorchKitServer` (`src/mcp/server.rs`)

```rust
pub struct ScorchKitServer {
    pub(crate) config: Arc<AppConfig>,   // Shared configuration
    pub(crate) pool: PgPool,             // PostgreSQL connection pool
    tool_router: ToolRouter<Self>,       // Generated by #[tool_router]
}
```

The server holds shared application state. The `tool_router` field is generated by rmcp's `#[tool_router]` macro from the annotated tool methods.

### `ServerHandler` Implementation

```rust
#[tool_handler]                          // Generates call_tool() + list_tools()
impl ServerHandler for ScorchKitServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo::new(
            ServerCapabilities::builder()
                .enable_tools()          // 24 MCP tools
                .enable_resources()      // 6 browsable resources
                .enable_prompts()        // 5 prompt templates
                .build()
        )
    }

    // Resource overrides (no macro — manual trait methods)
    async fn list_resources(...) { ... }
    async fn list_resource_templates(...) { ... }
    async fn read_resource(...) { ... }
}
```

**Key insight**: `#[tool_handler]` only generates `call_tool` and `list_tools`. Resource methods are overridden manually because rmcp has no `#[resource_handler]` macro. Both coexist in the same `impl ServerHandler` block without conflict.

### `do_*` Method Pattern (`src/mcp/tools.rs`, `src/mcp/resources.rs`)

All business logic lives in public `do_*` methods on `ScorchKitServer`:

```rust
// Public — called by tests directly
pub async fn do_project_list(&self) -> Result<String, String> { ... }

// Private — thin #[tool] wrapper for MCP dispatch
#[tool(description = "List all security assessment projects")]
async fn project_list(&self) -> Result<String, String> {
    self.do_project_list().await
}
```

This pattern keeps business logic testable without needing to pipe JSON-RPC through stdio.

### Parameter Types (`src/mcp/types.rs`)

Each tool has a corresponding params struct deriving `Deserialize` and `JsonSchema` (via `schemars`). rmcp uses the `JsonSchema` derive to auto-generate JSON Schema for MCP tool input validation.

```rust
#[derive(Debug, Deserialize, JsonSchema)]
pub struct ProjectScanParams {
    pub project: String,      // Project name or UUID
    pub target: String,       // Target URL
    #[serde(default = "default_profile")]
    pub profile: String,      // "quick", "standard", or "thorough"
}
```

## Resources (6 URIs + 5 Templates)

Resources provide read-only browsable access to project data. Unlike tools (which perform actions), resources let MCP clients discover and read data by URI without knowing which tool to call.

### URI Scheme

All resources use the `scorchkit://` protocol prefix with hierarchical paths matching the data model:

| URI | Returns |
|-----|---------|
| `scorchkit://projects` | JSON array of all projects |
| `scorchkit://projects/{id}` | Project details with targets, scan count, finding count |
| `scorchkit://projects/{id}/scans` | Scan history for a project |
| `scorchkit://projects/{id}/scans/{scan_id}` | Single scan record |
| `scorchkit://projects/{id}/findings` | All tracked findings for a project |
| `scorchkit://projects/{id}/findings/{finding_id}` | Single finding details |

### Resource Templates

Templates describe parameterized URI patterns for client-side URI construction:

| URI Template | Name |
|-------------|------|
| `scorchkit://projects/{project_id}` | Project Details |
| `scorchkit://projects/{project_id}/scans` | Project Scans |
| `scorchkit://projects/{project_id}/scans/{scan_id}` | Scan Details |
| `scorchkit://projects/{project_id}/findings` | Project Findings |
| `scorchkit://projects/{project_id}/findings/{finding_id}` | Finding Details |

### Implementation (`src/mcp/resources.rs`)

Resource URIs are parsed into a `ResourceKind` enum for type-safe dispatch:

```rust
enum ResourceKind {
    Projects,
    Project(Uuid),
    ProjectScans(Uuid),
    Scan(Uuid, Uuid),
    ProjectFindings(Uuid),
    Finding(Uuid, Uuid),
}
```

The `list_resources()` method is dynamic — it queries the database for current projects and returns a resource entry per project alongside the static `scorchkit://projects` collection. All resource content is returned as `application/json` text.

### No Subscription Support

ScorchKit is a CLI/MCP tool, not a long-running service. Data changes only when scans are explicitly triggered. Push notifications would add complexity with no benefit.

## Prompts (5)

Prompt templates give MCP clients structured starting points for common security assessment workflows. Exposed via `list_prompts` and `get_prompt` in the `ServerHandler` implementation.

### Implementation (`src/mcp/prompts.rs`)

Prompts are defined as compile-time constants in a `PROMPTS` array. The `ScorchKitServer` exposes `do_list_prompts()` and `do_get_prompt()` methods that map prompt names to multi-step workflow instructions. Each prompt returns a `GetPromptResult` with a system message guiding Claude through the appropriate tool sequence.

| Prompt | Description | Required Argument |
|--------|-------------|-------------------|
| `full-web-assessment` | Run a complete web application security assessment against a target | `target` (URL) |
| `investigate-finding` | Deep dive into a specific security finding with reproduction steps | `finding_id` (UUID) |
| `remediation-plan` | Generate a prioritized remediation plan for a project's findings | `project` (name or UUID) |
| `compare-scans` | Analyze what changed between two scans of the same target | `project` (name or UUID) |
| `executive-summary` | Generate a client-ready executive summary of a project's security posture | `project` (name or UUID) |

## Transport

### stdio (`scorchkit serve`)

The server communicates over stdin/stdout using rmcp's stdio transport:

```rust
pub async fn serve(config: Arc<AppConfig>) -> Result<()> {
    let pool = storage::connect_from_config(&config.database, None).await?;
    let server = ScorchKitServer::new(config, pool);
    let service = server.serve(stdio()).await?;
    service.waiting().await?;
    Ok(())
}
```

### Claude Code Integration

Add to Claude Code's MCP server configuration:

```json
{
  "mcpServers": {
    "scorchkit": {
      "command": "scorchkit",
      "args": ["serve"]
    }
  }
}
```

Claude can then use ScorchKit tools conversationally:

```
User: "Scan example.com for vulnerabilities"
Claude: [calls scan tool] → "Found 3 findings: 1 high (missing HSTS), 2 medium..."

User: "Create a project and track these"
Claude: [calls project_create, project_scan] → "Project 'example' created, scan persisted..."

User: "Show me the security posture"
Claude: [calls project_status] → "Trend: Improving. 3 findings, 0 regressions..."
```

## Feature Gating

```toml
[features]
default = []
storage = ["dep:sqlx", "dep:sha2", "dep:croner"]
mcp = ["storage", "dep:rmcp", "dep:schemars"]
```

The `mcp` feature implies `storage` because every MCP deployment needs persistence. The feature cascade:

| Feature | Adds | Modules |
|---------|------|---------|
| (default) | nothing | CLI scanner only |
| `storage` | sqlx, sha2, croner | + `src/storage/`, DB commands, scheduling |
| `mcp` | rmcp, schemars | + `src/mcp/`, `scorchkit serve` |

Conditional compilation in `lib.rs`:

```rust
#[cfg(feature = "mcp")]
pub mod mcp;
#[cfg(feature = "storage")]
pub mod storage;
```

## Testing Strategy

### Direct Method Testing

Tests call `do_*` public methods directly on `ScorchKitServer`, bypassing the stdio transport. This is simpler and more reliable than piping JSON-RPC since rmcp's transport layer is already well-tested.

```rust
#[tokio::test]
async fn test_tool_project_list() {
    let Some(pool) = get_pool_or_skip().await else { return };
    let server = test_server(pool);
    let result = server.do_project_list().await;
    assert!(result.is_ok());
}
```

### DATABASE_URL Skip Pattern

Integration tests that need PostgreSQL use a graceful skip pattern:

```rust
async fn get_pool_or_skip() -> Option<PgPool> {
    let Ok(url) = std::env::var("DATABASE_URL") else {
        eprintln!("DATABASE_URL not set -- skipping");
        return None;
    };
    // connect and migrate...
}
```

This means `cargo test` always passes — DB tests run when `DATABASE_URL` is set, skip gracefully otherwise.

## Dependencies

| Crate | Version | Purpose |
|-------|---------|---------|
| `rmcp` | 1.3 | MCP protocol: `ServerHandler` trait, `#[tool]`/`#[tool_router]` macros, stdio transport |

Both are feature-gated behind `mcp` and not included in the default build.

## Alternatives Considered

| Alternative | Rejected Because |
|-------------|-----------------|
| REST API (axum/actix) | Requires HTTP server, auth, CORS. MCP is the native AI protocol. |
| Hand-rolled JSON-RPC | Significant effort for dispatch, schema gen, transport. rmcp handles all of this. |
| gRPC (tonic) | Heavy toolchain (protobuf), not supported by Claude Code natively. |
| Always-on MCP (no feature gate) | Forces sqlx/rmcp on all users. CLI-only users don't need MCP overhead. |
| SSE transport | stdio is sufficient for Claude Code. SSE adds HTTP server complexity for remote use case not yet needed. |
| Resource macros | Don't exist in rmcp. Manual trait method override is the only option. |
| Subscription support | ScorchKit is not a long-running service. Data changes only on explicit scan triggers. |
